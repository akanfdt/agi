# CWM v1 Current Status

## Summary

This file is the current handoff note for the CWM project.

Current philosophy:

- CWM should learn like a small language universe
- We should avoid content-specific hacks
- Data cleaning and output-space definition are acceptable
- The "brain" should learn from experience rather than from fixed language rules

Current practical direction:

- Train on sentence content only
- Learn token flow, longer local paths, imitation, and compressed sentence memory
- Use sentence summaries as the center of understanding
- Keep testing with fresh initialization after meaningful code changes

## Current Architecture

- Public facade:
  - `cwm_core.py`
- Internal modules:
  - `cwm_types.py`
  - `cwm_points.py`
  - `cwm_gravity.py`
  - `cwm_context.py`
  - `cwm_orbit.py`

Tokenizer split:

- `tokenizer.py`
  - builds `cwm_tokens.json`
- `bpe_tokenizer.py`
  - runtime tokenizer implementation
  - used by training/runtime code

## What Is Implemented

### 1. Content-only training

Implemented in `cwm_train.py`.

Important helpers:

- `extract_content_text(...)`
- `iter_content_lines(...)`
- `iter_tokens_from_file(...)`

Meaning:

- speaker labels like `친구:` are stripped out
- base token training and dialogue/imitation both use sentence content only

This fixed the earlier speaker-label bias.

### 2. Modular CWM core

`CWMCore` is still the main entrypoint, but internal responsibilities are split.

Important public methods kept:

- `step_update(...)`
- `step_update_batch(...)`
- `advance_context(...)`
- `scores_from_context(...)`
- `predict_next(...)`
- `reinforce_dialogue(...)`
- `update_qa_memory(...)`
- `train_imitation_pair(...)`
- `save(...)`
- `load(...)`

### 3. Real imitation loop

Implemented in `cwm_core.py`.

Current imitation behavior:

- teacher-forcing style scoring over target tokens
- sequence imitation loss
- alignment tracking
- gravity/orbit reinforcement from imitation pairs
- vector nudging toward question context

This is real imitation-aware training, but it is still simpler than a full sequence model.

### 4. Longer local path learning

This was a major fix.

Old problem:

- base batch learning effectively reinforced mostly 2-token paths

Current fix:

- `step_update_batch(...)` now carries local recent tokens through the batch
- `_apply_learning_step(..., recent_override=...)` uses that local path
- `gravity.reinforce_sequence(...)` and `orbits.observe(...)` now see longer local token flows

This allowed 3- to 5-token local paths to start appearing more clearly.

### 5. Hierarchical sentence compression memory

Implemented in `cwm_core.py`.

Concept:

- recent sentences are kept in raw form
- older sentences are compressed into summaries
- compressed summaries are merged hierarchically over time

Current structures:

- `raw_line_memory`
- `compressed_line_memory`

Current default compression policy:

- keep 8 recent sentence summaries raw
- compress older summaries in pairs
- when compressed memory exceeds capacity, compress oldest compressed summaries again

This gives a lightweight long-context memory without directly linking every sentence to every other sentence.

### 6. Sentence-summary-centered design

This is the newest structural direction.

Implemented concepts:

- `sentence_summary_vector(tokens)`
- `summary_memory_vector()`
- `summary_prior()`
- `observe_sentence(tokens)`

Current idea:

- a sentence should first be compressed into an internal summary
- that summary should help shape later generation

### 7. Output token filtering

Implemented in `cwm_core.py` via `is_emittable_token(...)`.

Still intentionally kept:

- bare word-start marker is not emitted
- internal loop tokens are not emitted
- `<unk>` is not emitted

This is treated as output-space definition, not harmful language hardcoding.

## Current Important Experiments and Findings

### A. Empty output problem

Old issue:

- the model often emitted only the BPE word-start token
- decode then looked almost empty

Status:

- fixed by treating internal/meta tokens as non-emittable output tokens

### B. Speaker-label bias

Old issue:

- `친구:` and related dialogue labels dominated training

Status:

- fixed by content-only training

### C. 2-token orbit trap

Old issue:

- orbit learning was mostly stuck in short 2-token paths

Status:

- improved by passing batch-local recent token paths into learning

Evidence:

- 3- to 5-token orbit paths began appearing more clearly after the fix

### D. Importance too flat

Observed:

- very many tokens were receiving importance updates together
- top importance values became too flat and crowded

Root cause found:

- active set was effectively huge in practice
- too many tokens were rewarded every update

We then tried to narrow active candidates more aggressively.

### E. Importance too sharp

After aggressive narrowing:

- active set collapsed too hard
- nearly one token dominated each step
- outputs regressed toward repetitive `나는`, `맞아`, `네`, `?`, `!`

Conclusion:

- wide activation was bad
- ultra-narrow activation was also bad
- the system needs a middle state

### F. Score-scale instability

Observed:

- `gravity`, `transition`, `orbit`, and imitation terms could dominate semantic similarity
- this caused overconfident repetition

We tried:

- manual stabilization with fixed weights
- then strong adaptive auto-scaling

Result:

- fixed manual weighting was more stable
- strong adaptive auto-scaling over-amplified weak priors and caused collapse

Human analogy:

- quiet background thoughts were being amplified to the same volume as main thoughts

Conclusion:

- fully aggressive auto-scaling is not currently safe

### G. Summary as score bonus vs. summary as query center

We changed summary handling:

- before:
  - summary was one more score bonus term
- after:
  - summary is used to shape the query itself

Current helper structure:

- `_input_summary_vector(...)`
- `_context_summary_vector(...)`
- `_summary_core_vector(...)`

Meaning:

- model should first compress what it heard
- then speak from that compressed understanding

This direction is philosophically correct, but current summary quality is still weak.

## Current State of Hardcoding

Completely no hardcoding: false

Current status is:

- no content-specific word blacklist/ban logic
- no rule like "this exact word must never appear"
- but there are still structural heuristics and hand-set hyperparameters

Still intentionally present:

- content-only input cleaning
- non-emittable internal token filtering

Still structural/manual:

- activation control values
- importance selection values
- compression capacities
- summary decay / summary strength
- other numeric scaling and learning constants

Interpretation:

- content hardcoding is mostly absent
- structural/hyperparameter hardcoding still exists

## Fresh-Run Policy

This is important.

Current rule:

- after meaningful code changes, evaluate using fresh initialization
- do not trust old checkpoints when learning rules changed

Reason:

- old weights were trained under different "physics"
- mixing them with new code makes interpretation unreliable

## Current Best Understanding

The model is currently in a state like this:

- not empty anymore
- not just stuck in 2-token loops
- compressed sentence memory exists
- summary-centered understanding exists in structure
- but language generation is still immature

Typical remaining behavior:

- repetition of short high-frequency response habits
- weak true sentence-level coherence
- unstable balance between exploration and collapse

Human analogy:

- more like an early language learner than a mature speaker
- sometimes still falls back into a few safe repeated phrases

## Current Main Bottlenecks

### 1. Summary quality

The summary-centered design is correct, but current sentence summaries are still weak.

Problem:

- if the internal summary is weak, generation still falls back to familiar token habits

### 2. Stable mid-level attention

The system has already shown both extremes:

- too many candidates active
- too few candidates active

Need:

- a middle attention state

### 3. Output coherence

Even when the universe structure improves, output can still drift into:

- repeated short phrases
- punctuation loops
- local response habits

### 4. Automatic balancing

We tried replacing manual score weights with strong adaptive scaling.

Result:

- too aggressive
- weak priors got over-amplified

Future automatic balancing must be gentler and bounded.

- keep density-aware anchor spacing only at anchor birth time in `cwm_points.py`
- do not keep density repulsion on every vector update
- do not keep manual 1:1 current-vs-memory summary weighting
- do not keep the compressed-memory resolution-weighting experiment

### What was changed in code

#### 1. Sentence boundary preservation in base training

`cwm_train.py` was changed so base token-pair training no longer flattens the whole file into one continuous token stream.

Current behavior:

- iterate content line by content line
- tokenize each line separately
- build `(prev, tok)` pairs only within that line

Interpretation:

- this is treated as preserving natural sentence structure
- not as harmful hardcoding

Why it was changed:

- the previous trainer connected one sentence ending directly into the next sentence beginning
- this conflicted with the sentence-summary-centered philosophy

#### 2. Density-aware anchor birth in the point universe

`cwm_points.py` was changed so new anchors are born with local-density awareness.

Current behavior:

- when a new anchor is created near a base vector, the point store estimates a local neighbor center
- if the local region is crowded, the new anchor is nudged away from that local center
- this effect is used at anchor birth time only

Philosophy:

- user framed the model as a universe
- if stars are born too close together, old clusters over-dominate and new meaning collapses into them
- desired direction is natural spacing without brittle hard threshold rules

Important current decision:

- keep the birth-time spacing effect
- do not apply density repulsion during every normal vector update

Reason:

- repulsion on every update reduced crowding more strongly
- but it also damaged language structure and made output more fragmentary

### What was tested and what happened

All comparisons below were short fresh-run experiments using the same rough setup:

- about 3000 base-training tokens
- about 200 dialogue/imitation lines

#### A. Boundary-preserving baseline

Reference checkpoint:

- `tmp_boundary_check.pt`

Reference visualization:

- `cwm_universe_boundary_check.png`

Observed behavior:

- still collapsed into familiar response habits like `나는`, punctuation, and high-frequency fragments
- but this was the clean reference after fixing sentence-boundary handling

Human analogy:

- like an early learner who answers with a few familiar safe phrases

#### B. Manual equal weighting of old memory and current input

This was tried and rolled back.

What was tried:

- manually making current sentence summary and old summary memory speak with equal weight

Result:

- outputs became more fragmentary
- alignment degraded

Human analogy:

- old habits were quieted, but the current thought was still too weak to stand on its own

Conclusion:

- too manual
- rolled back

#### C. Density repulsion on every update

Reference checkpoint:

- `tmp_density_expansion_check.pt`

Reference visualization:

- `cwm_universe_density_expansion_check.png`

Result:

- crowding did decrease
- but language structure got worse
- output fell into even more fragmented punctuation/syllable behavior

Human analogy:

- like forcing a person to reorganize all memories every time they speak
- old rigidity weakens, but coherent thought weakens too

Conclusion:

- rolled back

#### D. Density repulsion only at anchor birth time

Reference checkpoint:

- `tmp_birth_only_expansion_check.pt`

Reference visualization:

- `cwm_universe_birth_only_expansion_check.png`

Result:

- better tradeoff than full repulsion
- some crowding reduction without the same degree of language collapse
- currently the best kept version from this session

Human analogy:

- let a new memory claim its own place when it is first formed
- do not force all existing memories to constantly spread apart

#### E. Reduced influence of deep compressed memory

Reference checkpoint:

- `tmp_birth_plus_resolution_memory.pt`

Reference visualization:

- `cwm_universe_birth_plus_resolution_memory.png`

What was tried:

- reduce the direct influence of old deep compressed summary segments

Result:

- did not improve short-run behavior enough
- alignment fell again in the short fresh-run comparison

Conclusion:

- rolled back

### Current best interpretation

The current model problem is not simply "old memory exists".

Better interpretation:

- old compressed memory can become too blurry and too habit-like
- new experience is then interpreted through that blurred background pressure
- if we reduce old pressure too manually, current understanding is still too weak and outputs fragment

Human analogy:

- not pure "brainwashing"
- more like strong old habits and blurred old recollections overpowering precise new understanding

### Important current caveats

- compressed memory can still become very coarse and very deep
- in a short fresh run we observed a compressed segment reaching `span=146`, `depth=73`
- summary memory still tends to favor broad frequent token pieces too much

### Recommended next direction after this handoff

- keep using fresh runs
- keep the sentence-boundary-preserving trainer change
- keep birth-time density-aware anchor spacing
- do not reintroduce manual ratio balancing unless the goal is diagnostic only
- if continuing the universe analogy, prefer local-density or age-based physics over explicit threshold/event hacks
- next promising focus is still summary quality and how current understanding can become strong without manual balancing
